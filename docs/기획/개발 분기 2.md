현재 계획

- 현재 실험 중단 -> 현재 파이프라인과 LLM 연동 -> Gemini로 객체 인식 보정 -> Gemini로 어포던스 및 실행 -> 유니티 개발 -> 파이프라인 완성

객체 인식 보정 및 어포던스 동시 추출을 위한 단일 VLM 기반 CoT 아키텍처 설계 - 의미 레이어(Semantic Layer) 재설계: VLM과 CoT를 활용한 어포던스 추출 및 지연 시간 최적화

- 실시간 AR 파이프라인에서 비전 모듈의 오탐지를 보정하고 어포던스를 추출하기 위해 Gemini(VLM)를 연동하는 과정에서 아키텍처 설계가 변경되었다.
- 초기에는 비전 파이프라인에서 추출한 기하학적 수치 데이터만을 JSON 형태로 LLM에 전송하여 논리적으로 오탐지를 보정하려고 했으나, 수치 데이터 자체가 부정확할 경우 한계가 있어 원본 카메라 이미지를 함께 전송하는 멀티모달 방식으로 전환하였다.
- 하지만 하나의 프롬프트에서 객체 오인식 보정과 어포던스 추출을 동시에 지시할 경우 작업 집중력이 희석되어 퀄리티가 떨어질 우려가 제기되었다. 이를 해결하기 위해 두 작업을 별개의 API 호출로 분리하는 방안을 고려했다. 그러나 어포던스라는 것은 사물이 놓인 주변 배경 환경에 대한 시각적 맥락이 반드시 필요하기 때문에, 만약 어포던스 추출 단계를 분리해서 텍스트만 주게 되면 배경 맥락을 잃게 되고 반대로 이미지를 두 번 주게 되면 실시간 AR 환경에서 지연 시간과 비용이 두 배로 증가하는 딜레마가 발생했다.
- 이러한 문제를 모두 해결하기 위해 최종적으로 단일 VLM 호출에 구조화된 Chain-of-Thought(CoT)를 적용하는 아키텍처를 채택했다. LLM이 한 번에 결론을 내놓아 집중력이 분산되는 것을 막고자, JSON 출력 스키마의 순서를 엄격하게 강제했다. LLM은 먼저 이미지의 주변 배경과 환경을 서술하고, 그 다음 객체의 진짜 정체성을 보정하고 물리적 상태를 판단한다. 그 후 앞서 파악한 배경과 상태를 종합하여 아바타가 할 수 있는 행동을 논리적으로 전개한 뒤 최종 어포던스 태그를 부여하게 된다.
- 더불어 API 호출 비용과 지연을 최소화하기 위해 매 프레임마다 이미지를 전송하는 대신 화면 내 객체의 개수나 면적, 깊이 등의 기하학적 수치가 크게 변했을 때만 이미지를 전송하여 결과를 갱신하는 동적 전송 로직도 함께 적용하기로 했다. 이 아키텍처를 통해 빠른 실시간성을 유지하면서도 배경을 고려한 정확한 객체 인지와 어포던스 판단이 모두 가능해질 것이다.

VLM CoT 아키텍처 실증 완료 및 지연 시간 최적화(Micro CoT) 설계안 - VLM 지연 시간 병목 해결: Micro CoT 도입 및 유니티 Action Planner 브리지 구축

- 의미 레이어(Semantic Layer)에 단일 VLM 기반의 CoT 아키텍처를 도입하여 실시간 테스트를 진행한 결과, 기하학적 데이터의 한계를 극복하고 시각적 상식을 통해 오탐지를 보정하는 연구 가설이 완벽하게 증명되었다. VLM은 단순 수치에 의존하지 않고 양손에 각각 들려있는 스마트폰과 휴대용 게임기를 정확히 구분해냈으며, 사용자의 상체와 손을 개별적으로 인지하여 들려있는 상태와 어포던스를 정확히 추론해냈다.
- 하지만 프레임당 11초에서 18초에 달하는 VLM 추론 지연 시간이 발생하여, 실시간 AR 환경 적용에는 심각한 병목으로 작용하는 문제가 발견되었다. 이 지연 시간을 1초에서 3초 사이로 대폭 단축하기 위해 초소형 생각의 사슬(Micro CoT) 전략과 전송 데이터 다이어트를 도입하기로 결정했다. LLM이 생성하는 배경 묘사와 행동 추론 과정을 장문이 아닌 단답형 혹은 15단어 이내의 짧은 문장으로 강력히 제한하여 출력 토큰 수를 최소화하고, 전송되는 이미지 해상도 역시 추가로 경량화하여 지연 시간을 극복할 계획이다.
- 더불어 이 의미 레이어의 결과를 다음 단계인 유니티(Unity)의 액션 플래너(Action Planner)로 원활하게 넘기기 위한 스키마 확장과 필터링 작업도 병행된다. 단순 자연어 추론을 넘어 유니티가 즉시 파싱할 수 있도록 어포던스 배열, 상호작용 가능 여부(is_interactable), 그리고 행동 정책(action_policy)과 같은 정형화된 필드들을 JSON 출력 스키마에 새로 추가한다.

ActionPlanner v1 정책 수정 및 의미 기반 공간 해석 검증 실험 계획서

1. 실험 목적

본 실험의 목적은 GPT 기반 Semantic Layer가 객체 정체성뿐 아니라 기하학적 위치 해석까지 의미론적으로 보정하고, 이를 기반으로 올바른 어포던스와 행동 정책을 생성할 수 있는지 검증하는 것이다.

기존 Geometry Layer는 객체의 좌표, 거리, 마스크 면적 등의 수치 정보를 계산할 수 있으나, 해당 수치로부터 도출되는 위치 해석은 항상 정확하지 않았다. 실제로 손에 들고 있는 객체를 바닥에 놓인 객체로 해석하거나, 전후관계 및 상하관계를 잘못 판단하는 사례가 확인되었다.

본 연구에서는 Geometry Layer가 계산한 수치 자체는 유지하되, GPT가 이미지와 기하학 정보를 함께 이용하여 객체 정체성, 공간적 위치 관계, 객체 상태를 재해석하도록 한다. 이후 보정된 공간 해석 결과를 기반으로 어포던스를 추출하고 ActionPlanner의 행동 결정을 수행한다.

1. 실험 환경

전체 파이프라인은 다음과 같다.

YOLO → SAM → Depth → Geometry Layer → GPT Semantic Layer → ActionPlanner → Unity Client

YOLO는 객체 후보를 탐지하고, SAM은 객체 마스크를 생성하며, Depth는 거리 정보를 계산한다. Geometry Layer는 좌표, 거리, 마스크 면적 등의 기하학적 수치를 계산한다.

GPT Semantic Layer는 원본 이미지와 Geometry Layer의 출력 정보를 입력으로 받아 다음 정보를 생성한다.

- object_identity
- corrected_spatial_relation
- object_state
- interaction_state
- affordances
- action_policy

여기서 corrected_spatial_relation은 Geometry Layer의 수치 정보를 기반으로 재해석된 위치 관계를 의미한다.

실험 대상 객체는 스마트폰, 텀블러, 닌텐도 3DS이며, 사용자는 해당 객체를 손에 들거나 카메라 앞에 배치한 상태에서 촬영을 수행한다.

1. 실험 가설

첫 번째 가설은 GPT가 객체 정체성을 의미론적으로 보정할 수 있다는 것이다.

두 번째 가설은 GPT가 Geometry Layer의 수치를 직접 수정하지 않더라도, 이미지와 수치 정보를 종합하여 잘못된 위치 해석을 보정할 수 있다는 것이다.

예를 들어 Geometry Layer가 객체를 on_floor로 해석하더라도 GPT는 이미지 맥락을 바탕으로 held_by_user 또는 on_surface로 재해석할 수 있다.

세 번째 가설은 보정된 공간 해석 결과를 기반으로 추출된 어포던스가 기존 규칙 기반 방식보다 더 자연스러운 행동 결정을 생성할 수 있다는 것이다.

1. 실험 절차

ActionPlanner v1 정책을 적용한다.

GPT Semantic Layer는 각 객체에 대해 객체 정체성, 공간 관계, 객체 상태, 상호작용 상태, 어포던스를 추론한다.

ActionPlanner는 GPT의 출력 결과를 기반으로 최종 행동을 결정한다.

사람(person)은 항상 상호작용 대상에서 제외한다.

interaction_state가 held_by_user 또는 currently_in_use인 경우 접근 행동을 수행하지 않는다.

object_state가 on_floor 또는 on_surface이고 interaction_state가 available인 경우에만 접근 및 상호작용을 허용한다.

실험 과정에서 다음 항목을 기록한다.

- object_identity
- corrected_spatial_relation
- object_state
- interaction_state
- affordances
- selected_action

특히 손에 들린 스마트폰, 텀블러, 닌텐도 3DS가 올바르게 held_by_user로 해석되는지, 그리고 ActionPlanner가 접근 행동을 선택하지 않는지를 중점적으로 확인한다.

1. 성공 기준

스마트폰, 텀블러, 닌텐도 3DS가 손에 들려 있는 경우 GPT가 이를 held_by_user 또는 이에 준하는 상태로 해석한다.

Geometry Layer의 초기 해석과 달리 GPT가 공간 관계를 의미론적으로 재해석하여 물리적으로 자연스러운 상태를 생성한다.

held_by_user 또는 currently_in_use 상태의 객체에 대해 ActionPlanner가 approach_interact를 선택하지 않는다.

person 객체는 항상 ignore로 처리된다.

GPT가 생성한 affordance와 ActionPlanner의 selected_action 사이에 논리적 일관성이 유지된다.

실험이 성공할 경우 GPT Semantic Layer가 단순 객체 분류 보정기가 아니라, 공간 관계 해석기 및 어포던스 생성기로 동작함을 확인한 것으로 판단한다. 이후 실제 Unity 환경에서 NavMeshAgent 이동 및 애니메이션 실행 단계로 확장한다.

1. 실험 환경

전체 파이프라인은 다음과 같이 구성한다.

YOLO → SAM → Depth → Geometry Layer → GPT Semantic Layer → ActionPlanner → Unity Client → NavMeshAgent → Avatar Animation

YOLO는 객체 후보를 탐지한다.

SAM은 객체 마스크를 생성한다.

Depth는 거리 정보를 계산한다.

Geometry Layer는 객체별 기하학적 원시 데이터(Bounding Box, Centroid, Depth, Mask Area)를 생성한다.

GPT Semantic Layer는 원본 이미지와 Geometry 정보를 이용하여 다음 정보를 생성한다.

- object_identity
- corrected_spatial_relation
- social_state
- affordances
- planner_directives

ActionPlanner는 GPT의 출력과 안전 규칙을 기반으로 최종 행동을 결정한다.

Unity Client는 WebSocket을 통해 ActionPlanner 결과를 수신하고, NavMeshAgent와 Animator를 이용하여 실제 아바타 행동을 수행한다.

첫 번째 가설은 GPT Semantic Layer가 생성한 의미 정보가 Unity 행동 제어에 직접 활용될 수 있다는 것이다.

두 번째 가설은 held_by_user 또는 currently_in_use 상태의 객체가 접근 대상에서 정상적으로 제외된다는 것이다.

세 번째 가설은 available 상태의 객체에 대해서만 아바타가 접근 행동을 수행한다는 것이다.

네 번째 가설은 GPT가 생성한 affordance와 Unity에서 실행되는 행동이 논리적으로 일관된다는 것이다.

다섯 번째 가설은 전체 파이프라인이 실시간 데이터 흐름을 유지하면서 Vision Layer부터 Avatar Action까지 연결될 수 있다는 것이다.

1. 실험 절차

먼저 Unity 3D 환경에 Plane, NavMesh, Avatar, NavMeshAgent를 구성한다.

Python 서버는 기존과 동일하게 WebSocket 서버를 유지한다.

Unity Client는 WebSocket을 통해 ActionPlanner 결과를 수신한다.

ActionPlanner 정책은 다음 규칙을 따른다.

- person → ignore
- held_by_user → ignore
- currently_in_use → ignore
- on_floor 또는 on_surface + available → approach_interact
- 그 외 상태 → ignore

Unity는 GPT가 생성한 과거 좌표를 직접 사용하지 않는다.

객체 식별자(object_id)를 기준으로 Geometry Layer의 최신 좌표를 추적하여 이동 목표를 갱신한다.

selected_action이 approach_interact인 경우 NavMeshAgent를 통해 목표 위치로 이동한다.

목표 지점 도달 시 affordance에 대응하는 애니메이션을 실행한다.

예시:

- observe → Observe Animation
- grasp → Grasp Animation
- drink → Drink Animation

실험 과정에서 다음 항목을 기록한다.

- object_identity
- corrected_spatial_relation
- social_state
- affordances
- selected_action
- avatar_behavior
1. 성공 기준

person 객체는 항상 ignore로 처리된다.

held_by_user 또는 currently_in_use 상태의 객체에 대해 Unity 아바타가 접근 행동을 수행하지 않는다.

available 상태의 객체에 대해서만 Unity 아바타가 접근 행동을 수행한다.

GPT가 생성한 affordance와 Unity에서 실행된 행동 사이에 논리적 일관성이 유지된다.

Python 파이프라인에서 생성된 의미 정보가 Unity 행동으로 정상적으로 전달되어 Vision → Semantic → Action → Avatar 실행의 전체 흐름이 관통된다.

본 실험이 성공할 경우 GPT Semantic Layer와 Unity Action Layer가 통합된 Embodied AI 파이프라인의 최소 기능 검증(Minimum Viable Embodied Pipeline)이 완료된 것으로 판단한다.

1. 프로젝트 개요

- 목표: 컴퓨터 비전과 대형 시각 언어 모델(VLM)을 결합하여, 가상 아바타가 주변 환경(Affordance)을 인지하고 자율적으로 행동하는 MVP 파이프라인 구축
- 구조: Python 기반 AI 비전 백엔드 서버 + Unity 3D 기반 프론트엔드 (WebSocket 양방향 실시간 통신)

2. 핵심 구현 내용

- VLM 기반 어포던스(Affordance) 인지 로직 구현
    - 객체의 단순 분류를 넘어 상태(available, held_by_user 등)를 파악하도록 프롬프트 설계
    - 상황에 맞춰 `APPROACH_AND_INTERACT` (접근), `IGNORE` (무시) 등의 자율 행동 정책 도출
- 비동기 기반 고속 실시간 스트림(Fast Stream) 구축
    - 무거운 LLM API 지연시간을 해결하기 위해 쓰레드를 분리하여 비동기 처리
    - 객체 좌표와 깊이 정보만 추출하여 유니티 클라이언트로 6FPS 초고속 브로드캐스트 전송
- Unity 3D 환경 제어 및 가상 AR 렌더링 도입
    - AR Foundation 등 무거운 프레임워크 없이 웹캠 화면을 Screen Space Camera 기법으로 렌더링
    - 파이썬의 2D 픽셀 및 깊이 좌표를 Unity NavMesh 3D 월드 좌표계로 정규화하여 매핑
    - 수신된 좌표를 바탕으로 `NavMeshAgent`를 활용한 아바타 자율 주행 로직 구현

3. 실험 결과 및 검증

- 방어 로직 작동: 웹캠에 사용자의 신체가 인식될 경우, VLM이 충돌 방지를 위해 즉각 `IGNORE` 명령을 하달하여 아바타 이동을 완벽히 차단함
- 타겟 추적 작동: 바닥의 스마트폰 객체 단독 인식 시, `APPROACH_AND_INTERACT` 정책이 수신되며 아바타가 3D 공간 상의 해당 좌표로 자율 주행하는 과정을 성공적으로 입증함

4. 향후 과제

- 애니메이션 연동: 현재의 좌표 기반 이동을 Unity Animator Controller와 결합하여 자연스러운 보행 애니메이션 구현 필요
- 카메라 캘리브레이션 정밀화: 실제 렌즈의 Intrinsic Matrix를 반영하여 센티미터 단위의 정밀한 AR 좌표계 변환 고도화 필요
